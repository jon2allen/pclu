# Compilation and 32-bit Linker Fix Report

This document summarizes the changes made to the `pclu` project to resolve linker errors, modernize C keyword usage, and ensure compatibility with modern 64-bit Linux and FreeBSD 15 environments via a 32-bit build path.

---

## 1. C Keyword Conflicts (`restrict`)
Modern C (C99 and later) introduced `restrict` as a reserved keyword. The `pclu` headers (originally written for older C) used `restrict` as a struct member name, which caused syntax errors in almost every translated C file.

- **File Modified**: `include/pclu_sys.h`
- **Change**: Renamed the `restrict` field in the `TG_ENTRY` structure to `restr`.
- **Impact**: Resolved hundreds of "expected identifier" and "invalid use of restrict" compilation errors across the codebase.

## 2. Linker Errors: Multiple Definitions
The project had several symbols defined in multiple places, which modern linkers reject by default.

### `GC_jmp_buf` (Garbage Collector)
The garbage collector failed to link because `GC_jmp_buf` was defined in both `gc_priv.h` and `os_dep.c`.
- **Files Modified**: 
  - `code/gc/include/private/gc_priv.h`: Changed the definition to `extern JMP_BUF GC_jmp_buf;`.
  - `code/gc/os_dep.c`: Added a single, guarded definition of `GC_jmp_buf`.

### Legacy Symbols (`TABLE`, `SELOPS`, `OPS`, `REQS`)
The machine-translated C code generated by the CLU compiler often defines common symbols like `TABLE` and `STR_` in multiple object files. 
- **Change**: Added the `-fcommon` flag to all `CFLAGS`. This tells GCC to merge multiple uninitialized global variables into a single definition.
- **Change**: Added `-Wl,--allow-multiple-definition` to the linker flags (`LFLAGS`) to allow the project to link even if initialized symbols are duplicated.

## 3. Migration to 32-bit (x86) Architecture
While initial fixes allowed the project to link in 64-bit mode, the resulting binaries were unstable. The CLU system makes hard-coded assumptions that a "word" is 4 bytes (e.g., `index << 2` for pointer arithmetic). In 64-bit mode, pointers are 8 bytes, causing array bounds failures and corruption when reading CLU library files.

### 3.1. Infrastructure Update
The `gcc-multilib` package (Linux) or `i386` libraries (FreeBSD) were installed on the host system to provide 32-bit development headers and libraries.

### 3.2. Build System Overhaul
All `Makefile.linux` files (11 in total) were updated to force 32-bit compilation and linking:
- **CFLAGS**: Prepend `-m32 -fcommon`.
- **LFLAGS / LDFLAGS**: Prepend `-m32`.
- **CC**: Updated to `gcc -m32` in the PCLU driver.

## 4. Cross-Platform Support (Linux & FreeBSD)
The codebase has been updated to support both Linux and FreeBSD 15 from the same branch.

### 4.1. Platform Detection
Modified the top-level `Makefile` to auto-detect the platform using `uname`:
```makefile
PLATFORM := $(shell uname -s | tr '[:upper:]' '[:lower:]')
```
This allows the same branch to compile correctly on both systems (finding `Makefile.linux` on Linux and `Makefile.freebsd` on FreeBSD).

### 4.2. Header and Function Adaptations
- **`dirent` Structure**: Replaced usage of non-standard `dp->d_namlen` with `strlen(dp->d_name)` in `_all_dir_entries_.c` for POSIX compliance.
- **File Closing**: Changed `close(dirp)` to `closedir(dirp)` in directory operations.
- **Signal Masks**: Added guards for signal mask structure differences (`__val` on Linux/glibc, `__bits` on FreeBSD).
- **Error Names**: Conditionally guarded `extern int sys_nerr` as it is handled differently by FreeBSD's headers.

### 4.3. Modern Compiler Flags
Added `-std=gnu89` and various `-Wno-error=...` flags to all Makefiles to satisfy modern `clang` and `gcc` strictness while maintaining support for the legacy generated C code.

---

## 5. Final Build Results
Following these changes, the project successfully completed a full build (`make veryclean && make`):
- **Garbage Collector**: Builds and passes all internal tests (`gctest`).
- **Compiler (`pclu`)**: Successfully links and is capable of compiling CLU source files.
- **Runtime Libraries**: `lowlev.lib`, `useful.lib`, and `misc.lib` are now correctly formatted and readable by the compiler.
- **Utilities**: `cludent` (the indenter) is built and operational.

## 6. Usage Note
To use the newly built system, you must define the `CLUHOME` environment variable:
```bash
export CLUHOME=$(pwd)
# Examples:
./exe/pclu                # Run the compiler
./exe/plink -o myprog...  # Link objects
```

## 7. Verification Example
To verify the system, a "Hello World" program with a loop was created and successfully compiled.

### Example Code (`hello.clu`)
```clu
% Hello World and for loop example in CLU

start_up = proc ()
    po: stream := stream$primary_output()
    
    % Print Hello World
    stream$putl(po, "Hello, World!")
    
    % Print numbers 1 to 20
    stream$putl(po, "Printing 1 to 20:")
    for i: int in int$from_to(1, 20) do
        stream$putl(po, int$unparse(i))
    end
end start_up
```

### Compilation Steps
1. **Set CLUHOME**:
   ```bash
   export CLUHOME=$(pwd)
   ```

2. **Compile to C & Object**:
   Use the `-opt` flag to avoid undefined tracing symbols (`CLU_TRACE`).
   ```bash
   ./exe/pclu -opt -spec lib/*.spc -co hello.clu
   ```

3. **Link**:
   ```bash
   ./exe/plink -opt -o hello_prog hello.o
   ```

4. **Execute**:
   ```bash
   ./hello_prog
   ```

## 8. Helper Script (`clu_compile.sh`)
To simplify the compilation process, a helper script `clu_compile.sh` was created. This script automates the steps of compiling and linking a single CLU source file.

### Usage
```bash
./clu_compile.sh [-o output_name] <filename.clu>
```
