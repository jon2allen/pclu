gh-Level Review and Action Plan for PCLU Build Failure on FreeBSD 15

## High-Level Review of the Issue

The immediate point of failure occurred when running `gmake veryclean` on FreeBSD 15. The top-level build system cascaded the `veryclean` target down into the `code/` subdirectory, which then attempted to call `make veryclean` on `code/gc` (the Boehm Garbage Collector version 7.2). Because Boehm GC uses a standard GNU Autotools build system (`configure`, `Makefile.in`), it supports standard targets like `clean`, `distclean`, and `maintainer-clean`, but not `veryclean`. This caused the make process to halt abruptly with an Error code 2.

Beyond the immediate clean failure, the log reveals a deeper, more systemic issue: **compiler strictness on modern operating systems**. The compiler is emitting hundreds of `-Wdeprecated-non-prototype` warnings. FreeBSD 15 uses a very modern compiler (Clang/LLVM 18+ or GCC 14+), which enforces strict C99/C11/C23 standards. Much of the C code—either generated by the CLU compiler (`cmp`) or handwritten in the PCLU runtime—relies on outdated K&R C styles (missing function prototypes, implicit return types) that are no longer supported in modern C23 standards.

Finally, when manually configuring the garbage collector, the script reports that **FreeBSD does not yet fully support threads with Boehm GC**, hinting at potential upcoming build or runtime problems with `gc-7.2` on FreeBSD 15.

---

### Potential Causes of the Failure

- **Incompatible Makefile Targets**: The custom PCLU `Makefile` in `code/Makefile` iterates blindly through all directories (including external dependencies like `gc`) and runs `make veryclean`.
- **Bit-rot against Modern C Standards**: PCLU’s C runtime heavily uses unprototyped functions. As compiler defaults shift towards C23, these turn from harmless warnings into fatal build errors.
- **Outdated External Dependencies**: Boehm GC 7.2 is quite old (circa 2012). The threading and memory layout semantics on modern FreeBSD 15 are vastly different, which can cause the `./configure` script to fail or build a broken garbage collector.

---

### Areas of Code to Investigate

- **`code/Makefile` (The Build System)**: The `veryclean` and `clean` rules in `code/Makefile` and the root `Makefile`.
- **PCLU C Headers and Runtime** (`code/include/`, `code/sysasm/Opt/`, `code/libasm/Opt/`): Files like `pclu_err.h`, `stream.c`, and `file_name.c` are throwing numerous warnings. These need to be investigated to see if fixing the headers to use ANSI C prototypes is feasible.
- **The CLU-to-C Compiler** (`code/cmp/`): Investigate if the C code generated by the CLU compiler uses K&R syntax. If so, the compiler itself needs patching.
- **Boehm GC configuration** (`code/gc-7.2/`): Investigate if PCLU truly relies on POSIX threads within the garbage collector.
